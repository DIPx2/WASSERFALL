ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ)
На разработку системы автоматизации управления PostgreSQL «Wasserfall»
================================================================================

1. ОБЩИЕ СВЕДЕНИЯ
--------------------------------------------------------------------------------
1.1. Наименование системы: Wasserfall: PostgreSQL Orchestration Engine.
1.2. Назначение: Инструмент для централизованного, параллельного выполнения 
     SQL-запросов на парке серверов PostgreSQL через SSH-туннели.
1.3. Принцип работы: Data-Driven Automation. Конфигурация подключений, 
     переменные окружения и шаблоны SQL-запросов хранятся во внешней базе 
     данных SQLite.
1.4. Язык реализации: Python 3.8+ (использование f-strings, type hints, pathlib).

2. АРХИТЕКТУРА СИСТЕМЫ
--------------------------------------------------------------------------------
Система представляет собой CLI-приложение модульной структуры.

+----------------------+---------------------------+--------------------------+
| Модуль               | Файл                      | Назначение               |
+----------------------+---------------------------+--------------------------+
| Entry Point          | main.py                   | Точка входа. Парсинг     |
|                      |                           | аргументов, оркестрация  |
|                      |                           | потоков.                 |
+----------------------+---------------------------+--------------------------+
| Config Provider      | modules/getter.py         | Извлечение конфигурации  |
|                      |                           | хостов, SSH-ключей,      |
|                      |                           | переменных и SQL-шаблонов|
|                      |                           | из SQLite. Управление    |
|                      |                           | SSH-сессиями (paramiko). |
+----------------------+---------------------------+--------------------------+
| Template Engine      | modules/template_engine.py| Рендеринг SQL-шаблонов   |
|                      |                           | через движок Jinja2 с    |
|                      |                           | подстановкой переменных. |
+----------------------+---------------------------+--------------------------+
| Executor             | modules/db_postgres_runner| Выполнение SQL на        |

3. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ
--------------------------------------------------------------------------------
3.1. Управление конфигурацией
    * Система должна читать настройки из файла databases/wasserfall_config.db.
    * Поддержка выборки хостов: все активные (toggle=1) или конкретный по имени.
    * Поддержка переменных окружения для SSH и PostgreSQL, хранящихся в БД 
      (таблицы ssh_variables, postgre_variables).
    * Поддержка хранения SQL-шаблонов в БД (таблица sql_commands).

3.2. Управление соединениями (SSH)
    * Использование библиотеки paramiko для SSH-соединений.
    * Поддержка типов ключей: Ed25519, ECDSA, RSA.
    * Обработка зашифрованных приватных ключей (запрос passphrase через 
      getpass при необходимости).
    * Политика управления известными хостами: AutoAddPolicy (разрешено) или 
      RejectPolicy (запрещено) в зависимости от флага запуска.
    * Таймауты соединения должны настраиваться (аргумент CLI).

3.3. Выполнение SQL-запросов
    * Рендеринг: Перед выполнением SQL-шаблон должен быть обработан движком 
      Jinja2 (StrictUndefined).
    * Транспорт: Выполнение команд через SSH с переключением пользователя на 
      системного postgres (su - postgres -c ...).
    * Формат результата: Исходный SQL-запрос автоматически оборачивается в 
      конструкцию SELECT json_agg(t) FROM (...) t; для получения валидного JSON.
    * Параллелизм: Выполнение задач по хостам должно осуществляться параллельно 
      с использованием ThreadPoolExecutor. Количество воркеров настраивается 
      (default=5).
    * Обход баз данных: Возможность получения списка пользовательских баз 
      данных через pg_stat_database и выполнение запроса по каждой (если не 
      указано явно через CLI).

3.4. Логирование и Аудит
    * Все выполнения задач должны фиксироваться в databases/wasserfall_logger.db.
    * Структура лога: ID задачи, Хост, Текст запроса, Код статуса PostgreSQL, 
      Код выхода, stdout (JSON), stderr.
    * Логирование должно быть асинхронным или не блокировать основной поток 
      выполнения критически.

4. МОДЕЛЬ ДАННЫХ (SQLite)
--------------------------------------------------------------------------------
4.1. Конфигурационная БД (wasserfall_config.db)
    1. hosts: id (PK), name (uniq), toggle (int 0/1).
    2. ssh_variables: id (PK), id_host (FK), variable, value.
    3. postgre_variables: id (PK), id_host (FK), variable, value.
       - Обязательные переменные: PG_DB_USER, PG_DB_PORT, PG_PSQL_PATH.
       - Опциональные: PG_DB_DEFAULT, PG_PASSWORD (если используется).
    4. sql_commands: id (PK), name (uniq), template (text).

4.2. БД Логирования (wasserfall_logger.db)
    1. execution_tasks: id (PK), target_host, query_text, created_at.
    2. execution_results: id (PK), task_id (FK), pg_code, exit_code, 
       stdout_json, stderr_text.

6. ОБРАБОТКА ОШИБОК И КОДЫ СТАТУСОВ
--------------------------------------------------------------------------------
Система должна унифицировать коды ошибок для последующего мониторинга.

6.1. SSH Ошибки (ssh_XX)
    * ssh_0  : Успех.
    * ssh_10 : Файл ключа не найден.
    * ssh_12 : Ключ зашифрован, ввод passphrase запрещен.
    * ssh_14 : Неверный passphrase.
    * ssh_16 : Не поддерживаемый формат ключа.
    * ssh_20 : Ошибка аутентификации или сети.
    * ssh_21 : Хост отсутствует в known_hosts.
    * ssh_22 : mismatch ключа хоста.
    * ssh_99 : Прочие ошибки.

6.2. PostgreSQL Ошибки (pg_XX)
    * pg_0  : Успех.
    * pg_10 : psql не найден (command not found).
    * pg_12 : Ошибка пароля (fatal ... password).
    * pg_14 : Синтаксическая ошибка SQL.
    * pg_16 : Ошибка подключения (could not connect, no route).
    * pg_18 : Ошибка выполнения (exit_code != 0, не пойманная выше).
    * pg_99 : Успешный exit_code, но не JSON (fallback).
	
6.3. Обеспечить вызов модулей только после в случае успеха выполнения предыдущих модулей.
     - Успех, если модуль возращает 0 с соответствующим префксом.
	 - Обеспечить логирование в базу данных каждого этапа.

7. ТРЕБОВАНИЯ К БЕЗОПАСНОСТИ
--------------------------------------------------------------------------------
1. Привилегии: Выполнение команд происходит от имени пользователя postgres 
   на удаленном хосте (через su). Учетная запись SSH должна иметь права на su.
2. Валидация: Использование StrictUndefined в Jinja2 для предотвращения 
   выполнения шаблонов с missing переменными.
3. SQL Injection: Параметры шаблона должны передаваться через контекст Jinja2, 
   а не конкатенацией строк. Обертка json_agg должна корректно экранировать 
   вложенные запросы.

8. ЗАДАЧИ ПО ДОРАБОТКЕ (GAP ANALYSIS)
--------------------------------------------------------------------------------
На основе предоставленного кода необходимо выполнить следующие доработки:

1. Доработка main.py:
   - Реализовать основную логику оркестрации.
   - Исправить вызов get_parse_args() (присвоить результат в переменную args).
   - Внедрить ThreadPoolExecutor для параллельного обхода хостов.
   - Реализовать цикл: Получение хоста -> Получение SSH клиента -> Получение 
     списка БД -> Рендеринг SQL -> Выполнение -> Логирование.
   - Обеспечить корректное закрытие SSH-соединений после выполнения задач.

2. Исправление импортов:
   - В getter.py функция get_user_databases вызывает 
     run_postgres_command_over_ssh, но не импортирует её. Необходимо добавить 
     импорт из db_postgres_runner.

3. Обработка результатов:
   - В db_postgres_runner.py при успехе (exit_code == 0) происходит попытка 
     парсинга JSON. Если JSON не валиден, данные сохраняются как строка. 
     Необходимо стандартизировать формат ответа для logger.py.

4. Пути к файлам:
   - Унифицировать работу с путями через pathlib (в logger.py используется 
     Path(file), что может вызвать ошибку NameError, если __file__ не определен 
     корректно в контексте запуска). Использовать ROOT из main.py.
